# CC Daily — 夜チューニングの圧縮ルール（昇格済みの学び）

毎週日曜の夜TUNE（`TUNE.md` 手順7「圧縮する」）が、**2回以上くり返し出た弱点・申し送りだけ**を短いルールに昇格してここへ足す。毎晩のTUNE（手順1「見る」）と判断（手順2「分ける」）が最初に読み、効かせる。

**昇格の基準（厳しく）：** 同じ弱点が別の日に2回以上出た／別runで再利用できた／このリポ固有の罠で今後も役立つ。1回きりの偶然・古い号固有の事情・一般論は載せない。

## 2026-07-05 昇格分

- 妻（yomimono_marks）は07-01のブートストラップ以降ずっとドキュメント0件が継続（Firestore取得自体はHTTP 200で成功＝取得失敗ではない）。5晩連続で同じ状況が出ているので、**マーク自体が無い間はinterest_tagsの並べ替え・増減提案を一切試みず、幅広い提示を維持したまま気長に待つ**。flaky/environmentどちらの分類でも対応は同じ（記録のみ）。
- favoriteRate は14日ウィンドウの presented（分母）が号の生成有無で日々変動するため、★の絶対数・updateTimeが変わっていないのに率だけ微増減することがある（07-03→07-05でpresentedが62→65、favoriteRateが0.1935→0.1846に見かけ上低下したが実際は無変化）。**率の上下だけで「上昇/下降」を判断せず、favoritedCountとタグ別countsの実数・★のupdateTimeが変化したかを必ず確認する。**

## 2026-07-12 昇格分

- `runTune.js` の `addCandidates`（新規タグ候補）は presented 窓内に同じ★が残っている限り連日出現し続ける。07-06に「AI活用」が初出現して以降、07-07〜07-12まで6晩連続で addCandidates に出ているが、中身は07-06に付いた同じ2件の★の使い回しで、その間新しい★は一度も付いていない。**addCandidates/newTags の複数日出現だけを再現とみなさず、★ドキュメントの updateTime が実際に更新されたか（＝新しい★が付いたか）を必ず確認してから deterministic 判定する。**
- 朝の号生成Routine（cc-daily trigger）は、`list_triggers` 上は毎朝正常に発火していても、記事収集・コミット段階のどこかで失敗し `issues/<date>.json` が生成されない日が起こりうる（07-04・07-07・07-09〜07-11で発生、07-12に復旧）。**Firestore取得の成否だけでなく `issues/<date>.json` の有無で当日号の生成を判定し、3日連続で欠落したら手順5のサーキットブレーカー基準通り人間にエスカレーションする。復旧を確認したら（該当コミットの存在で確認可能）その晩のうちにブレーカーを解除してよい。**
