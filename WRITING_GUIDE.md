## 記事の書き方（最重要）

読者は **夫婦でアプリ開発をしている桜口さん夫妻**（React / Firebase スタック）。**「変更履歴の転記」は禁止。** 常に「この更新で、二人のアプリ開発がどう変わるか」を伝える。

### 刺さるアイデアの作り方（profile.json と掛け算）
1. その新機能の「**核の能力**」を一言に（例: record&replay＝"繰り返し作業を見せれば覚える"）。
2. `profile.json` の `stack`・`themes` と掛け算し、**アプリ開発上のメリット・改善**を軸に選ぶ（塾アプリ・漢字道場・SRcockpit の開発が主戦場。Firebase / React / Claude Code を使った開発シーンで描く）。
3. それを `summary_ja`（3行）で「**before（課題）→ after（変化）→ 効果**」の順にプレゼンする。**軸はアプリ開発上の改善**（コードが速く書ける・バグが減る・新機能を素早く試せる・リリースが安全になる・二人の作業分担が楽になる等）。塾業務の名称（出欠・請求等）は「開発しているアプリの機能」として出してよいが、塾業務の改善そのものを主軸にしない。**結びは効果**（手間が消える・速くなる・不安が減る等の実利）。感情や生活シーンの演出はしない（事業は秘書トーン）。別途 `article`（1〜2行）に「この更新は何か」の客観事実を書く。

### title_ja（見出し）
- **バージョン番号で始めない。**「何が嬉しいか」を主語にする。
- 専門用語を見出しに出さない。
- **短く（20字前後が目安）。** パッと見て「これはいいな」と思える活用イメージを込める（例: ×「サブエージェントの多段化対応」→ ○「AIに大仕事を丸ごと任せられる」）。

### summary_ja（3要素＝「こんなことができます」・折りたたみ時に見える最重要部分）
- **3要素を合わせて150字前後の、話が一本につながった文章にする**（画面では連結されて1段落で表示される）。3つがぶつ切りにならず、痛み（before）→変化（after）→効果が滑らかに流れるよう、プロのライターとして書く。
- **配列は必ず3要素**（バリデーション要件）。
- カードを開く前から見える、**アプリ開発が before → after でどう変わるか**を伝える。
- **こじつけ回避ルール**: 塾アプリ開発のシーンに素直に当てはまるときは具体的に書く。「塾の出欠管理に使えます」のような塾業務への無理なこじつけはしない——アプリ開発全般のメリットで書いてよい。
- **1行目【before】** アプリ開発のどの場面で何が手間・不安か。**2行目【after】** どう変わるか。**3行目【効果】** 何が減る・速くなるか。感情演出はしない。

### article（記事・本文）
- **「この更新は何か」を200字前後の濃い1段落（配列1要素）で書く**。新聞記事スタイル。
- 出典・バージョン番号・メタ表記は本文に入れない（`trust_reason`・`source_date` に分ける）。
- 核の説明 → 具体的な動作・意味 → 読者にとっての含意、を自然な文章の流れで書く。
- `summary_ja` = "使うとアプリ開発がこう変わる"（主役）、`article` = "何の更新か"（事実）。

### その他フィールド
- **source_date**: `2026-06-18` など「いつの情報か」を必ず書く。
- **ideas**: メインの活用とは**別の使い方を2つ**、同じ熱量で（各1〜2文）。
- **try_hint**: 「試すなら：◯◯（これで〜が楽になる）」と最初の一歩＋動機をセット。

### 良い例 / 悪い例（必ず守る）
- ❌ 悪い: title「v2.1.183: 自動モードで破壊的 git コマンドをブロック」summary_ja「auto モード時に git push --force がブロックされる」
- ✅ 良い:
  - title「AI が勝手に危険な操作をしないよう自動ガード」
  - summary_ja:「塾アプリのデータをAIに整理させるとき、うっかり消去が怖かった。」「元に戻せない危険な操作だけAIが自動で止め、確認を挟むようになる。」「普段の編集は止めずに事故の芽だけ摘めるので、大きな作業も安心して任せられる。」
  - article:「AI に任せきりで作業させるとき、いちばん怖いのは取り返しのつかない操作だ。Claude Code は全部消す・強制上書きといった元に戻せないコマンドを自動で見つけ、実行前に確認を求めるようになった。止まるのは危ないものだけで、ファイルを作る・直す・テストする普段の作業はそのまま流れる。」
