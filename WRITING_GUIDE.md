## この記事は「教材」である（最重要）

読者は **アプリ開発をしている二人**（React / Firebase スタックの非エンジニア寄り）。求めているのは「最前線のエンジニアが実際にやっているテクニックを、自分でも使えるように学ぶこと」。
だからこの記事は**先生・コーチが教える教材**として書く。次の3つを必ず満たす：

1. **何が新しく加わった結果そうなったのか**を明示する（「6月のアップデートで〇〇が追加された」等）。ただの一般論にしない。
2. **覚えるべき具体的な名前（機能名・コマンド名・概念名）は消さずに残す**。それが学びの核。`fallbackModel`・`/cd`・`/rewind`・Dynamic Workflows 等は名前のまま出す。
3. **残した名前はその場で先生のように噛み砕く**。「`fallbackModel`（予備のAIを登録しておく設定）」のように、名前＋やさしい説明をセットにする。専門用語を裸で置かない。

### 一般的なアプリ開発の話として書く（固有名は出さない）
- **塾アプリ・漢字道場・SRcockpit・月謝・出欠などの固有名は出さない。** 「アプリ開発で〜」「開発中の機能で〜」と一般化する。
- 二人で開発しているという文脈はうっすら残してよい（「二人で別々の機能を並行して進める」等）が、特定の製品名・業務名には寄せない。

## 3つの欄の役割分担（重複させない）

| 欄 | 役割 | 書くこと |
|---|---|---|
| **こんなことができます**（summary_ja） | 結果 | あなたの作業がどう変わるか。専門用語ゼロ。状況→変化→効果 |
| **📰 記事**（article） | 事実・定義 | 何が新しく加わり、各機能は何か。覚える名前と定義。先生口調で噛み砕く |
| **🔧 試すなら**（try_hint） | 実践 | どう手を動かすか。手順つきで長めに。身につく知恵として |

**この3つは内容が被ってはいけない。** summary で機能名を定義し尽くすと記事と重複する。summary は「結果」、記事は「事実・定義」、試すなら は「手順」と切り分ける。

### title_ja（見出し）
- **ツール名を必ず入れる**（例: 「Claude Code の〜」）。タイトル単体で何の話か分かるようにする。
- **バージョン番号で始めない。**「何が嬉しいか」を主語にする。
- **短く（25字前後が目安）。** パッと見て「これは知りたい」と思える具体性を込める。

### summary_ja（＝こんなことができます・折りたたみ時に見える最重要部分）
- **3要素を合わせて150字前後、話が一本につながった文章**にする（画面では連結されて1段落で表示）。
- **配列は必ず3要素**（バリデーション要件）。
- **専門用語ゼロ。** 機能名の詳しい定義は書かない（それは記事の仕事）。ここは「あなたの作業がどう変わるか」の結果に集中する。
- **1行目【状況】** いまアプリ開発のどの場面で何が手間・不安か。**2行目【変化】** それがどう変わるか。**3行目【効果】** 何が減る・速くなるか。感情演出はしない。

### article（＝📰 記事・教材の核）
- **「何が新しく加わり、各機能は何か」を200字前後の濃い1段落（配列1要素）で書く**。新聞記事スタイル。
- **覚えるべき名前を必ず出し、その場で噛み砕く**（名前＋やさしい説明）。読者がこの段落だけで「何を覚えればいいか」を掴めるように。
- バージョン番号やメタ表記は本文に入れすぎない（`trust_reason`・`source_date` に分ける）が、「6月のアップデートで」など**いつ何が加わったか**は本文に書いてよい。

### try_hint（＝🔧 試すなら・実践レシピ）
- **手順つきで、やや長め**（2〜4文）。「まず〜して、次に〜すると、〜が起きる」と具体的な操作で導く。
- 読んだ人が「ちょっとやってみようかな」と思え、**一つの知恵として身につく**ことを狙う。
- 記事で出した名前（`fallbackModel` 等）を実際にどう使うかをここで見せる。

### その他フィールド
- **source_date**: `2026-06-18` など「いつの情報か」を必ず書く。
- **tags**: 「アプリ開発」「自動化」など2〜3語。

### 良い例 / 悪い例（必ず守る）
- ❌ 悪い（抽象的・用語なし・何が新しいか不明）:
  - title「自動で作業を続けさせる3つの設定」（ツール名なし）
  - article「AIが止まらず作業を続けてくれるようになり便利です」（覚える名前がない・何が加わったか不明）
- ✅ 良い（教材・名前を残して噛み砕く）:
  - title「Claude Code で自動作業を止めない、6月追加の3設定」
  - summary_ja:「AIに長い作業を任せても、途中でAIが混み合って止まり、そのたびに手で立て直していました。」「6月のアップデートで、こうした自動作業のつまずきを防ぐ3つの設定が加わりました。」「うまく使えば、夜のあいだ任せた長い作業も朝まで止まらず走り切らせやすくなります。」
  - article:「Claude Code の6月のアップデート（Week 24）で、自動作業を支える3つの機能が加わった。`fallbackModel` は、メインのAIが混み合って使えないときの代わりを最大3つまで登録しておく設定。`/cd` は、立ち上げ直さずに作業フォルダを切り替えるコマンド。`--safe-mode` は、自分で入れた設定を一時的に全部オフにして起動する方法で、不調のとき設定が原因かをすぐ切り分けられる。この3つの名前と役割が、長い自動作業を安定して走らせる土台になる。」
  - try_hint:「まず設定ファイルを開き、いつも使うAIの下に `fallbackModel` で予備のAIを1〜2つ書き足す（混んだとき上から順に試してくれる）。次に別プロジェクトを直したくなったら閉じずに `/cd フォルダ名` と打つと、その場で作業先が切り替わる。急に動きがおかしくなったら `--safe-mode` を付けて立ち上げ直すと、設定を全部オフにした素の状態で動くので原因の切り分けが一発でできる。」
